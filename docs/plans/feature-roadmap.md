## 기능 목록 및 구현 순서

### Phase 1: 홈서버 운영 필수 (Tier 1)

모니터 없는 서버에서 문제를 즉시 인지하고, 데이터를 보호하기 위한 최소 기능.

| ID | 기능 | 설명 | 변경 대상 | 의존성 |
|----|------|------|-----------|--------|
| T1-1 | Telegram 알림 | 체결/손절/에러/일일결산 알림 전송 | 신규: `src/notify/`, `.env` | 없음 |
| T1-2 | 헬스체크 API | FastAPI `/health` 엔드포인트 — 프로세스, DB, API 토큰 상태 | 신규: `src/api/health.py`, `main.py` | 없음 |
| T1-3 | DB 자동 백업 | pg_dump 일일 크론, 로컬 보관 (7일 롤링) | 신규: `scripts/backup_db.sh`, crontab | 없음 |
| T1-4 | 행(Hang) 감지 | 로그 미출력 감지 → 자동 재시작 | 신규: `scripts/watchdog.sh`, crontab | 없음 |

**구현 순서: T1-1 → T1-2 → T1-3 → T1-4**

- T1-1이 최우선: 이후 모든 기능의 에러/상태를 알림으로 받을 수 있음
- T1-2~T1-4는 독립적이므로 순서 유연

---

### Phase 2: 운영 편의성 (Tier 2)

폰/브라우저로 현황을 확인하고 원격 제어하는 기능.

| ID | 기능 | 설명 | 변경 대상 | 의존성 |
|----|------|------|-----------|--------|
| T2-1 | 웹 대시보드 | Streamlit — 보유종목, 당일 손익, API 호출량, 체결 내역 | 신규: `dashboard/` | T1-2 (헬스 데이터) |
| T2-2 | 성과 시각화 | 일별/주별/월별 수익률 차트, 종목별 승률, 전략 성과 | 신규: `dashboard/pages/` | T2-1 |
| T2-3 | 알림 레벨 분류 | 긴급(손절, 다운) → 즉시, 일반(체결, 결산) → 요약 | `src/notify/` | T1-1 |
| T2-4 | Telegram 원격 명령 | `/status`, `/stop`, `/balance`, `/today` 봇 명령 | `src/notify/` | T1-1 |

**구현 순서: T2-3 → T2-4 → T2-1 → T2-2**

- T2-3, T2-4는 T1-1(Telegram) 위에 확장하므로 먼저
- T2-1, T2-2는 별도 앱(Streamlit)이므로 독립 진행 가능

---

### Phase 3: 장기 개선 (Tier 3)

전략 고도화와 운영 자동화.

| ID | 기능 | 설명 | 변경 대상 | 의존성 |
|----|------|------|-----------|--------|
| T3-1 | 공휴일 자동 감지 | 한국 공휴일 API 연동, 휴장일 스케줄러 자동 스킵 | `src/scheduler/`, `src/config.py` | 없음 |
| T3-2 | 백테스팅 프레임워크 | 과거 데이터로 전략 시뮬레이션 | 신규: `src/backtest/` | 없음 |
| T3-3 | 다중 전략 동시 운영 | 종목별/상황별 전략 전환 | `src/strategy/`, `src/engine.py` | T3-2 (검증 필요) |
| T3-4 | 로그 구조화 저장 | 주요 이벤트를 DB 테이블로 저장, 조회 API | `src/db/models.py`, 신규 테이블 | T1-2 |
| T3-5 | Docker 컨테이너화 | docker-compose (앱 + PostgreSQL) | 신규: `Dockerfile`, `docker-compose.yml` | 모든 기능 안정화 후 |

**구현 순서: T3-1 → T3-4 → T3-2 → T3-3 → T3-5**

- T3-1은 간단하고 즉시 효과
- T3-5는 모든 기능 구현/안정화 후 마지막에 진행

## 기능 의존성 다이어그램

```
T1-1 Telegram 알림 ─────┬─→ T2-3 알림 레벨 분류
                         └─→ T2-4 원격 명령

T1-2 헬스체크 API ──────────→ T2-1 웹 대시보드 ──→ T2-2 성과 시각화

T1-3 DB 백업              (독립)
T1-4 행 감지              (독립)

T3-1 공휴일 감지          (독립)
T3-2 백테스팅 ────────────→ T3-3 다중 전략
T3-4 로그 구조화          (독립, T2-1과 시너지)
T3-5 Docker              (마지막)
```

## 안전 게이트 확장 필요사항

BRIDGE_SPEC.md에 추가해야 할 항목 (해당 기능 구현 시):

| 기능 | 안전 게이트 변경 |
|------|-----------------|
| T1-1 Telegram | `.env`에 `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID` 추가 (수동) |
| T1-2 헬스체크 | 새 파일 생성 허용 경로에 `src/api/health.py` 추가 |
| T2-1 대시보드 | `dashboard/`는 메인 앱과 별도 프로세스, BRIDGE_SPEC 대상 외 |
| T3-4 로그 구조화 | `alembic/versions/` 마이그레이션 필요 → 수동 검토 |
| T3-5 Docker | `Dockerfile`, `docker-compose.yml` 수동 검토 |

이렇게 되면 시간이 자산화됩니다. 내가 퇴근한 후에도 시스템이 진척을 만들어냅니다.

## 13. 만든 것 [1분 30초]

제가 만든 건 한국투자증권 OpenAPI 기반의 자동매매 시스템입니다.

장중에 자동으로 시세를 조회하고, 5종의 전략 -- 이동평균, RSI, MACD, 볼린저밴드, 그리고 이것들을 합친 앙상블 -- 으로 매매 판단을 합니다.

리스크 관리로 손절, 익절, MDD, 연패를 자동 감시하고, 종목 스크리닝으로 거래량 상위 종목을 자동 발굴합니다.

오른쪽을 보시면, 매일 16시에 Cowork가 분석하고, 17시에 Claude Code가 자동 구현하고, 금요일에는 주간 리뷰까지 합니다.

누적 자동 구현 63건, 테스트 427개, mypy strict 모드. 이 시스템을 가능하게 한 **자동화 패턴**에 집중하겠습니다.

## 16. 핵심 원리 2 -- 컨텍스트는 자산이다 [1분 30초]

두 번째, 컨텍스트를 문서로 관리합니다.

왼쪽 디렉토리 구조를 보시면, CLAUDE.md, BRIDGE_SPEC.md, reports, proposals가 있습니다.

CLAUDE.md는 프로젝트의 목적, 디렉토리 구조, 코딩 컨벤션, 모듈 경계, API 호출 정책이 들어있습니다. AI가 세션을 시작할 때 가장 먼저 읽는 파일입니다.

BRIDGE_SPEC.md는 Cowork와 Claude Code 사이의 자동화 규격입니다. 안전 게이트의 금지 영역, 25개 파라미터의 허용 범위가 여기에 명시되어 있습니다.

핵심은, AI에게 매번 컨텍스트를 말로 설명하지 않는다는 겁니다. 문서를 잘 정리해놓으면 AI는 알아서 그걸 읽습니다.

## 17. 핵심 원리 3 -- 로그가 아니라 데이터다 [1분]

세 번째, AI가 분석할 수 있는 형태로 데이터를 둡니다.

왼쪽은 Before입니다. 로그 파일에서 "지난주 손절 평균 손실률 알려줘"라고 AI에게 시키면 어떻게 됩니까? grep, 정규식으로 파싱하려다가 로그 포맷이 바뀌면 바로 깨집니다.

오른쪽은 After입니다. query_analytics.py를 돌리면 구조화된 JSON이 나옵니다. 손절 8건, 평균 -2.15%. 추천 손절률 0.0215.

Cowork가 이 JSON을 파싱해서 "손절률을 0.025로 조정하자"는 제안서를 자동으로 작성합니다.

판단 기준은 간단합니다. AI에게 분석을 시킬 때 grep이 필요하면, 데이터 구조를 다시 설계해야 합니다.

## 18. 핵심 원리 4 -- 안전 게이트 [1분 30초]

네 번째, 안전 게이트. AI가 만든 제안을 그대로 머지하지 않습니다.

제안서는 네 가지 상태로 흘러갑니다. ready에서 시작해서, 테스트가 통과하면 implemented, 실패하면 failed로 git restore, 안전 규칙을 위반하면 skipped.

아래쪽에 세 가지 규칙이 있습니다.

금지 영역 -- .env 수정, 인증 파일, 모의투자에서 실전으로 전환, 이런 건 절대 AI가 건드리지 못합니다.

파라미터 범위 -- 25개의 파라미터에 대해 허용 범위가 명시되어 있습니다. 예를 들어 MAX_LOSS_RATE은 0.01에서 0.05 사이만 가능합니다.

코드 변경 규칙 -- 제안서 하나당 최대 5개 파일, 파일 삭제 금지, 시그니처 변경 시 테스트 동반 필수.

"AI를 신뢰하느냐"는 잘못된 질문입니다. "코드로 검증되는 경계가 어디냐"가 올바른 질문입니다.

## 22. 패턴 2 -- 분석과 실행을 분리하라 [1분]

여러분 시스템에도 이런 데이터가 있을 겁니다. 운영 메트릭이든, 사용자 행동 로그든, 이슈 트래킹이든. AI가 질문할 수 있는 형태로 바꿔놓으면, AI 활용도가 완전히 달라집니다.

## 24. 패턴 4 -- AI 위임에는 검증 가능한 규칙이 있어야 한다 [1분]

네 번째 패턴.

"AI를 신뢰하는가"가 아니라, "코드로 검증되는 경계가 어디인가"를 물어야 합니다.

게이트를 설계할 때 네 가지를 정하면 됩니다.

무엇을 절대 안 건드리는가? 어디까지 자동으로 변경 가능한가? 검증은 어떻게 자동화되는가? 실패 시 어떻게 원복하는가?

이 네 가지만 명확하면, AI에게 안심하고 위임할 수 있습니다.

여러분 버전은 뭐가 될까요? 매일 배포 영향 분석, 매주 PR 트렌드 요약, 매월 보안 스캔.

## 28. 왜 Harness인가 -- 스크립트 기반의 한계 [1분 30초]

Harness 이야기를 하기 전에, 왜 다음 단계가 필요한지부터 말씀드리겠습니다.

지금 제 시스템은 잘 돌아갑니다. 그런데 한계가 있습니다.

역할 분리는 했지만, 그 분리를 제가 **쉘 스크립트**로 만들었습니다. run_daily_analysis.sh, run_auto_implement.sh. 에이전트끼리 통신은 proposals 디렉토리에 파일로 주고받습니다. 작업 순서는 launchd 시간표로 정해놨습니다. 16시 다음 17시.

이게 무슨 의미냐면, "AI를 운영"하지만 운영의 뼈대는 여전히 사람이 만든 스크립트라는 겁니다.

새로운 역할이 하나 추가될 때마다, 새 쉘 스크립트, 새 cron, 새 디렉토리 규약을 제가 만들어야 합니다. 그리고 한 번 실패하면 사람이 로그를 읽고 다음 cron까지 기다려야 합니다.

Harness는 이 천장을 뚫는 구조입니다. 역할은 마크다운 한 장으로, 통신은 에이전트 간 메시지 프로토콜로, 의존성은 시간이 아니라 태스크 그래프로 표현합니다.

한 줄로 정리하면, **Lv3는 AI를 부리는 사람이고, Harness는 AI 팀을 부리는 AI를 만드는 일**입니다.

## 29. Harness의 3계층 -- 누가 / 어떻게 / 언제 [2분]

Harness는 세 계층으로 분리되어 있습니다. 사람을 채용해서 팀을 만드는 과정과 똑같이 생각하시면 쉽습니다.

첫째, **Agent. "누가"** 합니다. 사람으로 치면 직무기술서입니다. .claude/agents 디렉토리 아래 market-analyst.md, code-implementer.md 같은 마크다운 파일로 존재합니다. 각 파일에는 그 에이전트의 정체성, 절대 원칙, 입출력 형식, 협업 관계가 적혀 있습니다.

둘째, **Skill. "어떻게"** 합니다. 사람으로 치면 업무 매뉴얼, SOP입니다. .claude/skills 디렉토리에 데이터 분석 방법, 안전 게이트 검증 절차 같은 게 들어있습니다. 중요한 점은 **점진적 정보 공개**입니다. 처음에는 짧은 요약만 로드되고, 필요할 때만 references 폴더의 상세 자료를 읽습니다. 컨텍스트 비용을 아끼기 위함입니다.

셋째, **Orchestrator. "언제"** 합니다. 사람으로 치면 PM, 팀장입니다. 작업을 분배하고, 병렬 가능한 것을 찾고, 결과를 모으고, 실패하면 폴백 경로를 결정합니다.

이 셋은 반드시 분리되어야 합니다. Agent에 Skill 내용을 다 박아넣으면 스킬을 다른 에이전트에서 재사용할 수 없습니다. Orchestrator에 역할을 박아넣으면 팀을 확장할 때마다 Orchestrator를 고쳐야 합니다.

## 30. Agent 파일 -- 어떻게 생겼나 [1분 30초]

실제 Agent 파일이 어떻게 생겼는지 보여드리겠습니다.

화면 왼쪽이 market-analyst.md 예시입니다. 코드가 아니라 **선언**입니다. 마크다운 한 장입니다.

맨 위 frontmatter에는 이름, 설명, 사용할 모델, 허용된 도구가 적혀 있습니다. description은 다른 에이전트가 이 에이전트를 호출해야 할지 자동 매칭할 때 사용됩니다.

그 아래로 정체성, 절대 원칙, 입력, 출력, 협업 섹션이 있습니다.

정체성은 "이 사람은 누구인가"입니다. 절대 원칙은 **이 에이전트가 절대 하지 않는 것**을 못 박습니다. CLAUDE.md의 전역 규칙을 이 역할에 맞게 좁히는 겁니다.

입력과 출력은 다른 에이전트와의 통신 프로토콜입니다. 파일 경로와 형식까지 명시해야 합니다. 그래야 뒷 에이전트가 받을 수 있습니다.

흥미로운 점은, 우리 프로젝트의 **BRIDGE_SPEC.md와 CLAUDE.md의 모듈 경계 표**를 그대로 에이전트 단위로 쪼개면, 이 파일들이 됩니다. 새로 쓰는 게 아니라 **이미 있는 컨텍스트를 재배치**하는 작업입니다.

## 31. Skill -- 점진적 정보 공개 [1분 30초]

스킬 디렉토리 구조를 보시면, 최상위에 skill.md가 있고, references 폴더와 scripts 폴더가 있습니다.

skill.md는 항상 로드됩니다. 짧게 씁니다. 트리거 조건과 핵심 원칙만 적습니다.

references 폴더는 **필요할 때만** 열립니다. 예를 들어 25개 파라미터 허용 범위 표는 매번 컨텍스트에 올릴 필요가 없습니다. "범위 검증이 필요할 때"만 references/ranges.md를 엽니다.

scripts 폴더는 결정적 도구입니다. LLM이 매번 새로 짜지 않고, 호출만 합니다. 검증 같은 결정적 로직을 코드로 격리하는 겁니다. **비결정성을 코드 바깥으로 빼는** 게 핵심입니다.

이 구조의 이득은 네 가지입니다.

컨텍스트 한계 우회. 재사용성 -- 같은 safety-gate 스킬을 code-implementer, qa-verifier, auto-heal이 공유합니다. 결정성. 그리고 버전 관리. 스킬도 PR로 리뷰합니다. "이 스킬이 어떻게 진화해왔나"가 git에 남습니다.

## 32. Orchestrator -- 4가지 협업 패턴 [1분 30초]

같은 에이전트들도, 어떻게 엮느냐에 따라 결과가 달라집니다. 네 가지 패턴이 있습니다.

첫째, **파이프라인**. analyst → implementer → qa. 앞 단계 결과가 뒷 단계 입력입니다. **지금 우리 시스템이 정확히 이 모양**입니다. 16시 분석, 17시 구현. 단계가 명확하고 의존성이 강할 때 씁니다.

둘째, **팬아웃 팬인**. 분석가 셋이 동시에 일하고, 통합자 한 명이 합칩니다. 시장 분석가, 리스크 분석가, 시그널 분석가가 병렬로 분석하고 synthesizer가 합치는 식입니다. 독립적 관점들이 모여야 할 때 씁니다.

셋째, **생성-검증**. implementer와 qa가 왕복합니다. 구현 → QA 지적 → 재구현 → 재검증. **실패가 학습이 됩니다**. 매치율 90%까지 자동 반복합니다. 한 번에 못 맞추는 작업에 적합합니다.

넷째, **감독자**. supervisor가 그때그때 적절한 에이전트를 선택합니다. 작업 구조가 사전에 결정되지 않을 때, 예를 들어 장애 상황에 적합합니다.

현실에서는 이게 섞입니다. 메인은 파이프라인, 분석 단계 안에서는 팬아웃, 구현은 생성-검증, 장애 시 감독자.

## 34. Harness가 가져올 변화 [1분]

Harness가 가져올 변화는 네 가지입니다.

전문성 분리 -- 분석, 구현, 검증 에이전트가 각자 깊이 있게 일합니다.
재사용성 -- 한번 정의한 에이전트와 스킬을 다른 프로젝트에서도 씁니다.
병렬 실행 -- 독립적인 작업은 동시에 처리해서 시간을 단축합니다.
품질 보장 -- QA 에이전트가 경계면을 교차 검증합니다.

Lv3의 다음 단계는, AI를 운영하는 시스템을 넘어서 **AI 팀을 설계하는 아키텍트**가 되는 것입니다.

## 35. KIS 프로젝트 장기 로드맵 -- Phase별 적용 [2분 30초]

그럼 우리 프로젝트에 실제로 어떻게 적용할 것인가. 4단계 로드맵입니다. **한 번에 다 바꾸지 않는** 게 핵심입니다.

**Phase 0, 현재**입니다. launchd 일곱 개, proposals 디렉토리, BRIDGE_SPEC.md가 사실상 에이전트 헌법 역할을 하고 있습니다. 자동 구현 누적 63건, 운영 비용 하루 5분. 잘 돌고 있습니다.

**Phase 1, 1~2개월**. 에이전트 분리, 즉 선언화입니다. BRIDGE_SPEC을 .claude/agents 네 개로 분해합니다. market-analyst, code-implementer, qa-verifier, auto-healer. safety-gate를 .claude/skills로 추출합니다. 중요한 건, **launchd는 그대로 둡니다**. 변화 최소화 원칙입니다. 이득은 새 에이전트, 예를 들어 보안 감사관 추가 비용이 마크다운 한 장으로 줄어든다는 점입니다.

**Phase 2, 3~4개월**. 팬아웃 분석, 즉 병렬화입니다. 16시 분석을 셋으로 쪼갭니다. 거래 패턴, 리스크와 MDD, 시그널 품질. synthesizer 에이전트가 통합 제안서를 만듭니다. 한 분석가가 못 보는 사각지대를 다관점으로 보완합니다.

**Phase 3, 5~6개월**. 생성-검증 루프. code-implementer와 qa-verifier가 자동 반복합니다. 지금은 pytest 한 번 실패하면 그날은 끝입니다. 앞으로는 실패 원인을 분석하고 재구현합니다. failed 비율이 줄고 사람 개입이 더 줄어듭니다.

**Phase 4, 6개월 이후**. 감독자 + 백테스트 통합입니다. supervisor가 일일 상황 보고 후 그날의 작업 구성을 동적으로 결정합니다. 새 전략 제안 → 자동 백테스트 → 통계적 유의미하면 가상계좌 카나리 → 점진 적용. 다만 안전장치는 더 보수적으로 갑니다. **실전계좌 변경은 여전히 사람의 영역**입니다. KIS_ENV가 real로 바뀌는 것은 자동화 대상에서 영구히 제외합니다.

로드맵 원칙은, 각 Phase는 이전 Phase 위에 얹는다는 것입니다. 실패 시 즉시 이전 Phase로 롤백 가능하도록 분기 단위로 작업합니다.

## 36. Phase 1 즉시 착수 가능한 작업 [1분 30초]

마지막으로, 다음 PDCA 사이클에서 바로 시작할 수 있는 가장 작은 첫걸음입니다.

화면 왼쪽이 디렉토리 구조입니다.

.claude/agents 아래에 네 개. market-analyst는 scripts/cowork_*.py와 BRIDGE_SPEC의 Cowork 부분을 옮깁니다. code-implementer는 run_auto_implement.sh와 안전 게이트 검증 로직을 가져옵니다. qa-verifier는 pytest, mypy, ruff 실행과 실패 진단을 분리해 받습니다. auto-healer는 scripts/auto_heal.sh의 판단 로직을 가져옵니다.

.claude/skills 아래에 셋. safety-gate에는 25개 파라미터 표와 금지 영역, 검증 스크립트가 들어갑니다. trading-analytics는 query_analytics.py 호출 SOP. log-diagnosis는 로그 패턴 분류 SOP입니다.

리스크를 분명히 합니다. **Phase 1은 리팩토링이지 신규 기능이 아닙니다**. 기존 동작이 그대로 보존되는지 회귀 테스트가 핵심 게이트입니다.

## 38. AI가 잘하는 것 / 사람이 잘해야 하는 것 [1분]

왼쪽은 AI가 잘하는 것입니다. 알려진 패턴의 코드 작성, 보일러플레이트, 테스트, 문서 초안, 데이터 분석, 리팩토링, 반복 자동화. 한 마디로 "전에 본 적 있는 일"을 빠르게 하는 겁니다.

## 부록. 예상 질문 (Q&A)

### Q1. Harness와 Subagents(서브에이전트)는 뭐가 다른가요?

좋은 질문입니다. 헷갈리기 쉬운 부분이라 정리해드리겠습니다.

**Subagents는 "기능"이고, Harness는 "아키텍처"입니다.**

Claude Code의 subagents는 도구입니다. `.claude/agents/` 아래에 마크다운으로 에이전트를 정의하면, 메인 세션이 필요할 때 Task 툴로 그 에이전트를 호출하는 방식입니다. 즉, **메인 Claude가 부하 직원을 잠깐 부르는 구조**입니다. 컨텍스트가 격리되고, 역할별로 도구 권한을 좁힐 수 있다는 장점이 있죠.

반면 Harness는 시스템 설계 패턴입니다. subagents를 포함해서 **Agent(누가) + Skill(어떻게) + Orchestrator(언제)** 세 계층을 어떻게 분리하고, 어떻게 협업 패턴(파이프라인 / 팬아웃 / 생성-검증 / 감독자)으로 엮을지에 대한 **설계 사상**입니다.

비유하자면, subagents는 "직원을 채용하는 기능"이고, Harness는 "조직도를 그리고 SOP를 정의하고 의사결정 라인을 만드는 일"입니다. subagents 없이도 Harness 사상은 적용할 수 있고(저처럼 launchd + 쉘 스크립트로), 반대로 subagents만 쓴다고 자동으로 Harness가 되는 것도 아닙니다.

핵심은, **subagents는 Harness를 구현하는 가장 자연스러운 1차 도구**라는 점입니다. Phase 1에서 BRIDGE_SPEC을 .claude/agents 네 개로 쪼개는 작업이 정확히 그겁니다.

### Q2. Harness를 처음 구성하려면 어떻게 시작해야 하나요?

세 단계로 권장드립니다. **한 번에 다 하지 마세요.**

**1단계, 컨텍스트 정리 (1주)**

먼저 CLAUDE.md를 정리합니다. 프로젝트의 목적, 디렉토리 구조, 모듈 경계, 절대 원칙을 적습니다. 이게 안 되어 있으면 어떤 에이전트도 깊이 있게 일을 못 합니다. **Harness의 토대는 코드가 아니라 문서**입니다.

**2단계, 역할 하나만 분리 (2~3주)**

전부 분리하려 하지 마세요. **가장 반복적이고 경계가 명확한 역할 하나**를 골라 `.claude/agents/{role}.md`로 빼냅니다. 저희 프로젝트라면 qa-verifier(pytest + mypy + ruff)가 1순위입니다. 입력 형식, 출력 형식, 절대 원칙, 사용 가능 도구를 frontmatter와 본문에 명시합니다.

이 단계에서 검증할 것은 두 가지입니다. **메인 세션과 컨텍스트가 깨끗하게 분리되는가**, **출력 포맷이 다음 단계가 받을 수 있는 형태인가**.

**3단계, Skill 추출과 협업 패턴 적용 (1~2개월)**

여러 에이전트가 공통으로 쓰는 절차(예: 안전 게이트 검증, 데이터 분석 SOP)를 `.claude/skills/`로 빼냅니다. 이때 **점진적 정보 공개** 원칙을 지킵니다. skill.md는 짧게, 상세 자료는 references/에, 결정적 로직은 scripts/에.

그 다음 협업 패턴을 정합니다. 처음에는 파이프라인(A → B → C) 하나면 충분합니다. 익숙해지면 분석 단계에 팬아웃을, 구현 단계에 생성-검증을 추가하는 식으로 확장합니다.

**시작할 때 피해야 할 함정 세 가지:**

첫째, Agent에 Skill 내용을 다 박아넣지 마세요. 재사용이 안 됩니다.
둘째, Orchestrator 없이 에이전트만 잔뜩 만들지 마세요. 호출 주체가 모호해집니다.
셋째, 검증 가능한 안전 게이트 없이 자동화부터 하지 마세요. 신뢰는 코드로 강제되어야 합니다.

가장 빠른 시작은, **이미 잘 돌고 있는 워크플로우 하나를 골라 그걸 Harness 구조로 리팩토링**하는 겁니다. 신규 기능으로 시작하면 변수가 너무 많아집니다.

### Q3. 운영 비용은 얼마나 드나요? AI API 비용이 부담되지 않나요?

솔직히 말씀드리면, 처음에는 저도 이 부분이 가장 걱정이었습니다.

현재 제 시스템은 **Claude Code Max 정액제**로 돌리고 있어서, 자동 구현/분석에 들어가는 LLM 비용은 사실상 정액 안에서 해결됩니다. 거기에 PostgreSQL, Redis, KIS API는 본인 계좌만 있으면 무료/저렴이고, **인프라는 맥북 한 대**입니다. 클라우드 비용 0원입니다.

만약 API 종량제로 환산한다면, 하루에 분석 1회 + 구현 1회 + auto-heal 몇 번 정도라 입력 토큰 대부분이 캐시 히트라서 생각보다 적게 나옵니다. 핵심은 **컨텍스트를 문서로 잘 정리해두면 캐시 적중률이 올라가서 비용이 떨어진다**는 점입니다. 컨텍스트 자산화는 비용 최적화이기도 합니다.

회사 도입을 검토하신다면, 한 명의 개발자 한 시간 인건비와 비교해보시면 답이 나올 거라고 생각합니다.

### Q4. 안전 게이트가 못 잡은 사고는 없었나요? AI가 잘못 짠 코드를 머지한 적은요?

있었습니다. 솔직하게 공유드리겠습니다.

가장 흔한 케이스는 **테스트는 통과했는데 의도가 틀린 경우**입니다. 예를 들어 Cowork가 "손절률을 0.025로 조정"하자고 제안했는데, Claude Code가 손절률은 잘 바꿨지만 관련된 백테스트 시나리오를 함께 갱신하지 않아서 다음날 백테스트 결과가 이상하게 나온 적이 있습니다. pytest는 그래도 통과한 거죠.

이런 사고가 발생할 때마다 **안전 게이트 규칙을 코드로 추가**합니다. "파라미터를 바꾸면 관련 테스트가 같은 PR에 포함되어야 한다" 같은 규칙이죠. 사고가 학습이 됩니다.

핵심은, **사고가 안 나는 시스템을 만드는 게 아니라, 사고가 나도 블래스트 반경이 작고, 빠르게 롤백되고, 같은 사고가 두 번 나지 않는 시스템**을 만드는 겁니다. 그래서 git restore, failed 상태, 그리고 KIS_ENV=real 영구 제외 같은 보수적 경계가 필수입니다.

### Q5. 팀에 도입하려고 하면 분명히 저항이 있을 것 같습니다. 어떻게 설득하시겠어요?

가장 현실적인 질문이네요.

저는 **"AI에게 일을 맡기자"는 프레임으로는 절대 설득이 안 된다**고 생각합니다. 사람들이 위협을 느끼니까요.

대신 **"우리가 매주 반복하는 그 짜증나는 작업, 그거 자동화하자"**로 접근합니다. 매주 월요일 PR 트렌드 정리, 매일 배포 영향 분석, 매월 보안 스캔 같은 거요. 아무도 좋아하지 않는 일부터 시작하면 저항이 거의 없습니다.

그리고 **첫 도입은 절대 크리티컬 패스에 두지 마세요**. 메인 빌드 파이프라인에 AI를 넣으면 한 번 사고나면 그걸로 끝입니다. 부가적인 리포트, 분석, 문서화부터 시작해서 신뢰를 쌓아갑니다.

마지막으로, **가시적인 산출물**이 중요합니다. 매주 자동 생성되는 리포트가 슬랙에 올라오기 시작하면, 사람들이 "어, 이거 편하네"라고 느낍니다. 그때부터 "이것도 자동화 가능해?"라는 질문이 들어옵니다. 그게 도입의 진짜 시작점입니다.

### Q6. 주니어 개발자는 이제 뭘 해야 하나요? AI가 코드를 다 짜면 학습 기회가 없을 텐데요.

이게 제일 어려운 질문입니다. 저도 정답은 모릅니다.

다만 분명한 건, **"AI가 짠 코드를 읽고 판단할 수 있는 능력"이 가장 중요한 기초 체력**이라는 점입니다. 코드를 처음부터 끝까지 손으로 짜본 경험이 없으면, AI가 만든 코드의 문제를 찾을 수 없습니다. 그래서 기본기는 오히려 더 중요해졌다고 봅니다.

그리고 **Lv1, Lv2, Lv3를 순서대로 밟아야 합니다**. Lv1에서 AI에게 물어보며 모르는 걸 채우고, Lv2에서 위임하며 시스템 전체를 보는 눈을 키우고, Lv3에서 설계합니다. 처음부터 Lv3로 못 갑니다.

마지막으로 한 가지 덧붙이면, **"AI가 못 하는 일"은 점점 줄어들고 있지만, "AI에게 무엇을 시킬지 정의하는 일"은 줄어들지 않습니다**. 문제 정의, 도메인 이해, 트레이드오프 판단. 이 영역에 일찍 발을 들이는 주니어가 다음 5년을 가져갈 거라고 생각합니다.

### Q7. 실제로 수익은 나고 있나요?

발표 주제와는 살짝 떨어진 질문이지만, 자주 받는 질문이라 답변드리겠습니다.

이 시스템의 **목적은 수익 극대화가 아니라 "AI 운영 시스템의 검증 환경"**입니다. 자동매매가 잘 돌면 수익이 따라오겠지만, 제가 오늘 보여드린 가치는 **"코드 변경 63건이 사람 손 없이 진행되는 파이프라인이 실제로 동작한다"**는 것입니다.

수익률 자체는 모의투자로 검증 중이고, 통계적으로 유의미한 결과가 나오기 전까지는 실전 전환 안 합니다. 그리고 발표 첫머리에서 말씀드린 대로, **실전계좌로의 전환은 자동화 대상에서 영구히 제외**입니다. 그건 사람의 영역입니다.

오늘 발표의 메시지는 "주식으로 돈 버는 법"이 아니라 "AI를 운영하는 사고방식"이었다는 점, 다시 한 번 강조드리고 싶습니다.

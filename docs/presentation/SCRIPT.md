## 9. Lv2. 위임하는 개발자 [1분 30초]

두 번째 단계. 위임하는 개발자.

Claude Code, Cursor 같은 에이전트 도구를 사용합니다.
이 도구들은 프로젝트의 코드를 알아서 읽습니다. MCP로 DB나 외부 시스템도 연결됩니다.

"이거 구현해줘" 하면 파일을 읽고, 수정하고, 테스트까지 만들어줍니다.

확실히 1단계보다 훨씬 빠릅니다.

하지만 한계가 있습니다. **여전히 사람이 트리거해야 합니다.**
"오늘 뭘 개선할까"는 사람이 결정합니다. 분석과 구현이 한 세션에 뒤섞입니다.
그리고 반복적인 분석 작업은 여전히 수동입니다.

요즘 일 잘하는 개발자들이 도달해 있는 단계라고 봅니다.

## 10. Lv3. 설계하는 개발자 [1분 30초]

세 번째 단계. 설계하는 개발자.

여기서 결정적인 차이가 생깁니다.

분석하는 AI와 구현하는 AI를 다른 세션으로 분리합니다.
그리고 **사람이 아니라 스케줄러가 AI를 호출합니다.**

컨텍스트는 문서로 관리해서 AI가 스스로 읽게 합니다.
AI의 제안에는 자동 안전 게이트가 적용됩니다.

사람은 뭘 합니까? 이 시스템을 설계하고, 결과를 보고, 의사결정만 합니다.

이렇게 되면 시간이 자산화됩니다. 내가 퇴근한 후에도 시스템이 진척을 만들어냅니다.

오늘 이 발표가 여러분을 이 단계까지 데려오는 것이 목표입니다.

## 13. 만든 것 [1분 30초]

제가 만든 건 한국투자증권 OpenAPI 기반의 자동매매 시스템입니다.

장중에 자동으로 시세를 조회하고, 5종의 전략 -- 이동평균, RSI, MACD, 볼린저밴드, 그리고 이것들을 합친 앙상블 -- 으로 매매 판단을 합니다.

리스크 관리로 손절, 익절, MDD, 연패를 자동 감시하고, 종목 스크리닝으로 거래량 상위 종목을 자동 발굴합니다.

하지만 오늘 이야기의 핵심은 매매 전략이 아닙니다.

오른쪽을 보시면, 매일 16시에 Cowork가 분석하고, 17시에 Claude Code가 자동 구현하고, 금요일에는 주간 리뷰까지 합니다.

누적 자동 구현 63건, 테스트 427개, mypy strict 모드. 이 시스템을 가능하게 한 **자동화 패턴**에 집중하겠습니다.

그리고 watchdog이 5분마다 프로세스를 감시하고, 반복 장애가 발생하면 auto_heal이 Claude Code를 호출해서 자동 진단과 수정까지 합니다.

## 15. 핵심 원리 1 -- 역할 분리 [1분 30초]

자, 이제 이 시스템을 가능하게 한 핵심 원리 네 가지를 설명합니다.

첫 번째, 역할 분리. 하나의 AI에게 모든 걸 시키지 않습니다.

왼쪽이 Cowork입니다. 분석과 제안을 담당합니다. 평일 16시에 깨어나서 DB를 쿼리하고, 리포트를 쓰고, 제안서를 만듭니다. 코드는 절대 안 건드립니다.

오른쪽이 Claude Code입니다. 구현과 테스트를 담당합니다. 17시에 깨어나서 ready 상태 제안서를 받아서 구현합니다. 분석은 안 합니다.

왜 이렇게 했냐면, 분석할 때 필요한 컨텍스트와 코드를 짤 때 필요한 컨텍스트가 다르기 때문입니다. 한 세션에 섞으면 둘 다 얕아집니다.

역할이 다르면 세션도 달라야 합니다.

## 16. 핵심 원리 2 -- 컨텍스트는 자산이다 [1분 30초]

두 번째, 컨텍스트를 문서로 관리합니다.

왼쪽 디렉토리 구조를 보시면, CLAUDE.md, BRIDGE_SPEC.md, reports, proposals가 있습니다.

CLAUDE.md는 프로젝트의 목적, 디렉토리 구조, 코딩 컨벤션, 모듈 경계, API 호출 정책이 들어있습니다. AI가 세션을 시작할 때 가장 먼저 읽는 파일입니다.

BRIDGE_SPEC.md는 Cowork와 Claude Code 사이의 자동화 규격입니다. 안전 게이트의 금지 영역, 25개 파라미터의 허용 범위가 여기에 명시되어 있습니다.

핵심은, AI에게 매번 컨텍스트를 말로 설명하지 않는다는 겁니다. 문서를 잘 정리해놓으면 AI는 알아서 그걸 읽습니다.

이 문서들은 코드와 함께 git으로 버전 관리됩니다.

## 17. 핵심 원리 3 -- 로그가 아니라 데이터다 [1분]

세 번째, AI가 분석할 수 있는 형태로 데이터를 둡니다.

왼쪽은 Before입니다. 로그 파일에서 "지난주 손절 평균 손실률 알려줘"라고 AI에게 시키면 어떻게 됩니까? grep, 정규식으로 파싱하려다가 로그 포맷이 바뀌면 바로 깨집니다.

오른쪽은 After입니다. query_analytics.py를 돌리면 구조화된 JSON이 나옵니다. 손절 8건, 평균 -2.15%. 추천 손절률 0.0215.

Cowork가 이 JSON을 파싱해서 "손절률을 0.025로 조정하자"는 제안서를 자동으로 작성합니다.

판단 기준은 간단합니다. AI에게 분석을 시킬 때 grep이 필요하면, 데이터 구조를 다시 설계해야 합니다.

## 18. 핵심 원리 4 -- 안전 게이트 [1분 30초]

네 번째, 안전 게이트. AI가 만든 제안을 그대로 머지하지 않습니다.

제안서는 네 가지 상태로 흘러갑니다. ready에서 시작해서, 테스트가 통과하면 implemented, 실패하면 failed로 git restore, 안전 규칙을 위반하면 skipped.

아래쪽에 세 가지 규칙이 있습니다.

금지 영역 -- .env 수정, 인증 파일, 모의투자에서 실전으로 전환, 이런 건 절대 AI가 건드리지 못합니다.

파라미터 범위 -- 25개의 파라미터에 대해 허용 범위가 명시되어 있습니다. 예를 들어 MAX_LOSS_RATE은 0.01에서 0.05 사이만 가능합니다.

코드 변경 규칙 -- 제안서 하나당 최대 5개 파일, 파일 삭제 금지, 시그니처 변경 시 테스트 동반 필수.

"AI를 신뢰하느냐"는 잘못된 질문입니다. "코드로 검증되는 경계가 어디냐"가 올바른 질문입니다.

## 22. 패턴 2 -- 분석과 실행을 분리하라 [1분]

두 번째 패턴. 역할이 다른 AI 세션은 다른 사람으로 대하라.

안티패턴을 보시면, "우리 시스템 분석하고 문제 찾고 코드 고치고 테스트도 만들어줘" -- 이렇게 한 세션에서 다 시키면 어느 것도 깊이 있게 못 합니다.

한 명에게 다 시키지 마세요. 역할을 나누세요.

여러분 시스템에도 이런 데이터가 있을 겁니다. 운영 메트릭이든, 사용자 행동 로그든, 이슈 트래킹이든. AI가 질문할 수 있는 형태로 바꿔놓으면, AI 활용도가 완전히 달라집니다.

## 24. 패턴 4 -- AI 위임에는 검증 가능한 규칙이 있어야 한다 [1분]

네 번째 패턴.

"AI를 신뢰하는가"가 아니라, "코드로 검증되는 경계가 어디인가"를 물어야 합니다.

게이트를 설계할 때 네 가지를 정하면 됩니다.

무엇을 절대 안 건드리는가? 어디까지 자동으로 변경 가능한가? 검증은 어떻게 자동화되는가? 실패 시 어떻게 원복하는가?

이 네 가지만 명확하면, AI에게 안심하고 위임할 수 있습니다.

## 25. 패턴 5 -- 반복 루틴은 스케줄러에 넘겨라 [1분]

이건 지금 제 맥북에서 실제로 돌고 있는 7개의 launchd 서비스입니다.

매매 엔진, 프로세스 감시, 일일 분석, 주간 리뷰, 자동 구현, 대시보드, DB 백업. 전부 스케줄러가 깨웁니다.

여러분 버전은 뭐가 될까요? 매일 배포 영향 분석, 매주 PR 트렌드 요약, 매월 보안 스캔.

"AI 써야지" 하고 사람이 매번 트리거하면 결국 안 쓰게 됩니다. AI는 스케줄러가 깨워야 합니다.

## 28. Harness란 무엇인가 [1분 30초]

Harness는 AI 에이전트를 팀으로 운영하는 아키텍처입니다.

세 가지 구성 요소가 있습니다.

Agent는 "누가" 합니다. 분석가, 구현자, QA 검증자. 각각의 역할과 원칙, 통신 규칙이 마크다운 파일로 정의됩니다.

Skill은 "어떻게" 합니다. 데이터 분석 방법, 안전 게이트 규칙, 진단 방법. 에이전트가 참조하는 구체적인 작업 수행 방법입니다.

Orchestrator는 "언제" 합니다. 작업을 분배하고, 병렬로 실행 가능한 것을 식별하고, 결과를 수집하고, 에러 시 폴백 경로를 결정합니다.

---

## 29. 우리 시스템에 Harness를 적용한다면 [1분]

현재 제 시스템은 왼쪽처럼 스크립트 기반입니다. launchd가 시간대별로 각각의 쉘 스크립트를 호출하고, 에이전트 간 통신은 proposals 디렉토리의 파일로 합니다.

Harness를 적용하면 오른쪽처럼 됩니다. 에이전트가 직접 통신하고, 작업 의존성을 관리하고, 팀 단위로 조율합니다.

핵심 차이는, 현재는 "사람이 스크립트로 파이프라인을 짜는" 구조인데, Harness에서는 "AI가 AI 팀을 조율하는" 구조가 됩니다. 사람은 팀 구성만 설계합니다.

## 30. Harness가 가져올 변화 [1분]

Harness가 가져올 변화는 네 가지입니다.

전문성 분리 -- 분석, 구현, 검증 에이전트가 각자 깊이 있게 일합니다.
재사용성 -- 한번 정의한 에이전트와 스킬을 다른 프로젝트에서도 씁니다.
병렬 실행 -- 독립적인 작업은 동시에 처리해서 시간을 단축합니다.
품질 보장 -- QA 에이전트가 경계면을 교차 검증합니다.

오른쪽에 네 가지 아키텍처 패턴이 있는데, 파이프라인, 팬아웃/팬인, 생성-검증, 감독자. 상황에 따라 조합합니다.

Lv3의 다음 단계는, AI를 운영하는 시스템을 넘어서 AI 팀을 설계하는 아키텍트가 되는 것입니다.

## 32. AI가 잘하는 것 / 사람이 잘해야 하는 것 [1분]

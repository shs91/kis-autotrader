### 3-1. 앙상블 서브전략 분석

이번 주 15,918건의 SIGNAL_SKIP 기록을 분석한 결과, 7개 관심종목에 대해 4개 서브전략 모두가 예외 없이 HOLD(confidence=0.0)를 반환했다.

| 전략 | 시그널 | confidence | 비고 |
|------|--------|------------|------|
| 이동평균교차(5/20) | HOLD | 0.0 | 교차 미발생 (정상 계산, guard 미작동) |
| RSI(14) | HOLD | 0.0 | 정상 범위 35~70 (과매도 35 설정 적용 중) |
| MACD(12,26,9) | HOLD | 0.0 | MACD-Signal 교차 미발생 |
| 볼린저(20,2.0) | HOLD | 0.0 | 가격이 밴드 내부 |

`guard_triggered` 필드가 null인 점에서 데이터 부족이나 NaN 문제는 아닌 것으로 판단된다. 4/20 일봉 조회량 수정(150일분)이 정상 적용되어 `series_len` 충분, 각 전략이 정상 계산 후 교차/극단값 조건에 해당하지 않아 HOLD를 반환한 것이다.

**근본 원인: 일봉 캐시 + 앙상블 HOLD 과반 가드의 구조적 조합**

1. **일봉 캐시**: 일봉 데이터는 하루 1회만 갱신된다(`_daily_cache`). 장중 1,500~2,200 사이클을 실행해도 전략 입력 데이터는 동일하므로 첫 사이클의 결과가 종일 반복된다.
2. **HOLD 과반 가드** (4/17 구현): 4개 전략 중 과반(3개 이상)이 HOLD이면 앙상블도 HOLD을 반환한다. 일봉 기반 교차 전략은 대부분의 거래일에 교차가 없으므로, 4개 전략 중 3~4개가 HOLD인 날이 대다수다.
3. **BUY 시그널 발생 조건**: MA 교차 + RSI 과매도 + MACD 교차 + BB 이탈 중 최소 3개가 동시에 발생해야 BUY가 가능한데, 이는 현실적으로 극히 드문 시장 조건이다.

### 3-5. 스크리닝 효율

주간 평균 전환율: **0.0%** (9일 연속 0%). 4/22 ScreeningWorker 메서드명 버그 수정 이후에도 전환율이 복구되지 않았다. 이는 메서드 호출 자체의 문제가 아니라, 전략 분석 결과가 모두 HOLD이기 때문이다.

스크리닝 스코어링 구조:
```
total_score = volume_rank × 0.2 + change_rate × 0.3 + strategy × 0.5
```
- strategy_score는 `BUY 시그널이 아니면 0.0`
- 현재 전략이 BUY를 전혀 생성하지 않으므로 strategy_score는 항상 0
- 최대 가능 점수 = 0.2 + 0.3 = 0.5 (strategy 기여 없이)
- min_score = 0.25이므로 volume+change_rate만으로도 일부 통과 가능하나, 실제로는 `rank_candidates → new_codes`가 비어 전환되지 않음

## 6. 중기 아키텍처 논의

### 6-1. 매매 교착의 구조적 원인

현재 시스템의 매매 파이프라인에는 구조적 병목이 있다:

1. **일봉 캐시 + 고빈도 사이클의 모순**: 일봉 데이터는 하루 1회 갱신되는데 매매 사이클은 하루 1,500~2,200회 실행된다. 첫 사이클 이후의 모든 사이클은 동일한 결과를 재계산할 뿐이다. 이는 API 호출 낭비(현재가 조회만 반복)이자 불필요한 시스템 부하다.

2. **앙상블 합의 요구 수준**: 4개 교차/극단값 전략의 과반 합의를 요구하면 BUY 시그널 발생 빈도가 극히 낮아진다. 일봉 기반 MA 교차는 평균 수십 거래일에 1회, RSI 과매도는 극단적 하락장에서만, MACD 교차도 수일~수주 간격으로 발생한다. 이들이 동시에 일치할 확률은 현실적으로 월 1~2회 이하다.

3. **스크리닝→전략→매매 전환율의 연쇄 단절**: 스크리닝이 수백 종목을 발굴해도, 전략 분석에서 BUY가 나오지 않으면 전환율은 0%다. 스크리닝의 volume/change_rate 필터링은 작동하지만, 최종 관문인 전략 스코어(가중치 50%)가 항상 0이므로 사실상 전략이 스크리닝을 무력화하고 있다.

### 6-2. 비동기 병렬 처리 필요성

사이클 시작/완료 차이가 주간 내내 0~1건으로, 현재 시스템은 병목 없이 순차 처리를 소화하고 있다. API 한도 도달도 0회. **현 단계에서 비동기 병렬 처리의 필요성은 낮다.**

### 6-3. 전략 앙상블 구조 재검토 필요성

현재 앙상블은 "합의에 의한 보수적 매매"를 지향하나, 결과적으로 "합의 불가에 의한 매매 정지"가 되었다. 두 가지 개선 방향이 있다:

**방향 A — 앙상블 유지, 투표 기준 완화**:
- HOLD 과반 가드의 임계값을 50%에서 75%로 상향 (4개 중 3개 이상이 HOLD일 때만 앙상블 HOLD)
- 또는 BUY 시그널 1개 + 나머지 HOLD(반대 없음)이면 약한 BUY로 통과

**방향 B — 개별 전략 독립 실행**:
- 각 전략의 시그널을 독립적으로 DB에 기록하고, 가장 높은 confidence의 BUY 시그널을 채택
- 앙상블은 분석 도구로만 유지하고 실제 매매 결정은 개별 전략에 위임

두 방향 모두 데이터 축적 후 백테스팅이 선행되어야 한다. 현재 4주간 매도 4건으로는 백테스팅의 기준선(baseline)조차 확보되지 않은 상태이므로, 우선 **매매 빈도 자체를 회복하는 것이 1순위**다.

### 6-4. 백테스팅 프레임워크 필요성

누적 손익 데이터가 4주간 6거래일분밖에 없어 패턴 분석이 불가능하다. 백테스팅 프레임워크(`src/backtest/`)가 존재하지만, 현재 전략 파라미터 변경의 효과를 사전 검증할 수 있는 파이프라인이 자동화되어 있지 않다.

제안: 전략 파라미터 변경 전에 최근 90일 일봉 데이터로 백테스트를 실행하여 시그널 빈도와 예상 수익률을 검증하는 워크플로우를 자동 파이프라인에 추가. 단, 이는 대규모 변경이므로 단계별 분할이 필요하다.

4. **사이클 주기 최적화** — 일봉 캐시 기반에서는 사이클 간격을 현재의 ~10초에서 60~120초로 늘려도 동일한 매매 결과를 얻을 수 있다. API 호출 절감과 로그/메트릭 볼륨 축소 효과.

5. **백테스트 자동화** — 전략 파라미터 변경 제안서에 "최근 90일 백테스트 결과" 첨부를 필수화.

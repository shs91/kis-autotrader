## 현상 분석

스크리닝이 정상 동작하여 `screening_results` 테이블에 일 129종목(283사이클)을 적재하지만,
엔진의 `EVAL_TARGETS` 메트릭에서 `screening: 0`이 **종일 지속**(1,184건 전수 확인).

```
EVAL_TARGETS 예시:
{"cycle": 1111, "counts": {"positions": 0, "screening": 0, "watchlist": 7}, ...}
```

결과적으로 watchlist 7종목만 평가 대상이 되어 시그널 0건, 매매 0건 교착.

### 근거 데이터
- 쿼리 6: 오늘 스크리닝 129종목, 전환 0건 (0.0%)
- 쿼리 11: 최근 7일(4/22~4/28) 전환율 **전일 0.0%**
- EVAL_TARGETS: 1,184건 전수 `screening: 0`
- 쿼리 9: 사이클 1,187회 정상 완료 (완료율 99.9%)

### 기존 시도 이력
4/22~4/27 구현된 7건(파라미터 튜닝, 진단 로깅 등) 모두 이 근본 원인을 해결하지 못함.
`screening_results` 적재는 정상이므로 스크리닝 자체는 동작. **엔진이 스크리닝 결과를 조회하는 경로**에 문제 존재.

## 제안 내용

`src/engine.py`에서 매매 사이클 실행 시 `screening_results`로부터 최근 스크리닝 종목을 조회하여
평가 대상(`targets`)에 포함하는 로직을 점검·수정한다.

가능한 원인:
1. 스크리닝 결과 조회 쿼리가 빈 결과를 반환 (조건 불일치, 시간 범위 등)
2. 스크리닝 Worker의 출력을 엔진이 읽는 인터페이스 불일치 (메서드명, 반환 타입 등)
3. 스크리닝 결과 → 엔진 평가 대상 변환 로직의 예외 무시(silent catch)

## 변경 스펙

### 파일별 변경사항
- `src/engine.py`: 스크리닝 결과 조회 로직 디버깅 및 수정
  - `_get_evaluation_targets()` 또는 유사 메서드에서 `screening_results` 조회 경로 확인
  - 스크리닝 종목이 0건인 경우의 로그 강화 (조회 SQL, 조건, 결과 건수)
  - 조회 조건 수정 (시간 범위, cycle_number 매칭 등)

### 추가 테스트 (필요 시)
- 스크리닝 결과가 존재할 때 평가 대상에 포함되는지 단위 테스트

## 기대 효과

`EVAL_TARGETS.screening > 0` 복원 → 스크리닝 종목 시그널 평가 활성화 → 매매 교착 해소.
129종목이 평가 대상에 추가되면 시그널 발생 가능성이 현재(watchlist 7종) 대비 대폭 증가.
